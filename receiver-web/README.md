# AirScan-QR 浏览器接收端

## 打开方式

双击 `receiver.html`，推荐使用 Chrome 或 Edge 120 及以上版本。页面不需要
Python、本地服务或安装程序。

## 接收文字

文字传输完成后会显示在页面顶部，“最新收到的文字”始终排在最前方。每条消息都可以
单独复制，页面也会尝试把新消息写入系统剪贴板；浏览器拒绝自动写入时不影响消息展示。

文字消息只保留在当前页面会话中，刷新或关闭页面后会清空，不写入 IndexedDB，也不会
下载为 `message.txt`。

## 接收文件

1. 保持桌面发送端窗口可见，不要最小化。
2. 浏览器页面默认选择 `1x1`，发送端使用其他宫格时在页面中同步选择。
3. 点击“选择共享窗口”，在浏览器权限窗口中选择发送端窗口。
4. 页面识别到任务后会显示文件名、接收进度和缺失帧数。
5. 全部帧通过 SHA-1 校验后，浏览器自动开始下载。

文件保存到浏览器配置的默认下载目录。浏览器安全策略不允许本地网页静默指定
其他目录；是否弹出保存位置由浏览器的下载设置决定。

共享画面位于页面底部并仅显示缩略图。扫码使用隐藏 Canvas 中的视频原始分辨率，
缩略图尺寸不会降低 `1x1`、`2x2` 或 `3x3` 的解码输入质量。

## 暂停和补发

“暂停扫描”只停止画面识别，不会清除 IndexedDB 中已接收的分片。重新打开页面时，
最近的未完成任务会自动恢复。点击“查看缺失序号”可以复制 `181, 283, 546-559`
格式的序号，并粘贴到桌面发送端的“缺失序号补发”输入框。

## 解码器加载

页面优先从 jsDelivr 加载固定版本 jsQR 1.4.0；网络不可用时自动使用
`vendor/jsQR.js` 本地副本。协议解析、分片存储和文件校验始终在本机浏览器内完成。

## 识别优化说明

网页端使用 jsQR，能力弱于桌面端 ZBar，因此做了速度优先的识别链路：

- 小预览定位二维码亮区，再裁剪解码（避免 2K/4K 整帧硬扫）
- 裁剪后缩放到 jsQR 友好尺寸，失败时再做亮度拉伸
- 共享优先选“窗口”并请求较高分辨率；视频轨 `contentHint=detail`

**识别率关键指标**：页面右下角 FPS。正常应 ≥ 5；若只有 0.x，说明上一帧解码过慢、会漏掉发送端循环帧。

## 常见问题

- 浏览器没有出现共享选择框：确认由按钮点击触发，并使用最新版 Chrome 或 Edge。
- 一直未识别 / 不如桌面端：
  1. 共享时只选 **AirScan-QR 发送端窗口**，不要共享整个桌面或浏览器大窗。
  2. 发送端用 **1×1**、纠错 **M 或 L**，窗口尽量放大，二维码占满可见区域。
  3. 硬刷新接收页（Ctrl+F5）后看 FPS 是否回到 5+。
  4. 网页端对高密度二进制 QR 的上限仍低于桌面 pyzbar/ZBar，大文件更建议桌面接收。
- 共享突然结束：重新点击“重新选择共享窗口”，现有任务进度不会丢失。
- SHA-1 校验失败：任务会清空损坏分片并保留元信息，继续扫描或让发送端重新广播。
